# keiba ハーネス不変則（正本）

> このファイルは `.claude/rules/` 内なので**セッション起動時に自動ロード**される。
> ハーネスの**強制ルールの正本**＝CLAUDE.md や各 reference の方針ブロックと食い違ったら**ここが勝つ**。
> 不変則を変えるときは**まずここを直し**、各文書のエコーを追従させる（手順は [editing-map](editing-map.md)）。

レポート（`/analyze-race`）・ログ・採点（`/review-prediction`）すべてに常時かかる。番号は優先度でなく参照用。

## I1. 市場ゼロ（純粋情報の原則）＝**予想ハーネス（`/analyze-race`）に適用**
- オッズ・人気・他人の予想（専門紙の印・予想展開記事）は**証拠にもログにも一切使わない**。
- **買い目・EV・Kelly・妙味・期待値・回収率は出さない**。馬券選択・実ベットは**人間判断**。
- 予測は馬の内在情報のみで完結させる。
- 適用範囲は**予想本体**＝`/analyze-race` とその成果物（report.json／predictions.jsonl／research-*.json）。**選別レイヤーは I1-S、結果層の配当記録は I1-R、EV提示は I1-E の隔離条件下でのみ市場を扱う**。いずれも report.json／predictions.jsonl／research-*.json には一切流さない。

## I1-S. 選別レイヤーの例外（市場の隔離・`/screen-card`）
- 勝負レース選別ハーネス（`/screen-card`）は**予想とは別レイヤー**として、レース選別の目的に限り**市場（単勝オッズ）を読んでよい**（団子度＝Y軸の当日側）。
- **隔離の壁（破ってはならない）**:
  1. オッズ取得（`tools/fetch_odds.py`）の出力は **`data/screening/` にのみ**書く。`research-*.json`／`report.json`／`predictions.jsonl` には**一切流さない**。
  2. `/analyze-race`（予想本体）は従来どおりオッズを**一切見ない**＝X軸（誰が勝つか）は内在情報だけで作る。選別の団子度は X の証拠にしない。
  3. 選別の出力は**レース単位の妙味判定まで**（🟥本命勝負／🟦鉄板コツコツ／見送り）。**馬ごとの買い目・金額・EV・賭け金は出さない**（I1 の「買い目は人間」は選別でも維持。EV は選別の出力に含めない＝EV は別レイヤー **I1-E** の成果物）。
- 思想: 市場は「賭ける瞬間だけの隔離レイヤー」。純度は予想側で守られる。

## I1-R. 結果層の配当記録（払戻＝確定公示事実・箱ROI検証用）
- レース確定後の**払戻（配当）は市場予想値でなく公示された事実**として、`data/results.jsonl` に `record:"payout"` 行（1レース1行・追記のみ・100円あたり円建て・的中組番）で記録してよい。
- **隔離の壁（破ってはならない）**:
  1. 配当は**確定後のみ**記録する。予想成果物3種（report.json／predictions.jsonl／research-*.json）には一切流さない＝予想成果物の禁止語スキャン（validate_report／validate_research_bundle／assemble_report／risk_flags）は**現状維持**。
  2. 用途は**箱戦略バックテスト（`tools/box_sim.py`）と EV の事後検証に限る**。観点の証拠・読み筋の根拠には使わない。
  3. `/review-prediction` の採点（2スコアカード）に**回収率を混ぜない**＝検証対象は依然「展開精度・着順精度」。箱ROIは box_sim の**別トラック**。
  4. オッズ・人気は payout 行にも記録しない（記録するのは払戻金額と的中組番の事実のみ）。
- 思想: 「予想が当たるか」（精度＝I4 の2スコアカード）と「箱が儲かるか」（ROI＝I1-R トラック）を**別々に測る**。前者が汚れなければ後者を測ってよい。

## I1-E. EVレイヤー（`/ev-board`・買う瞬間の隔離判断板）
- EV提示ハーネス（`/ev-board`）は**予想とも選別とも別レイヤー**として、馬連・三連複オッズを読み、箱候補ごとの EV（的中確率×オッズ）を提示してよい。
- **隔離の壁（破ってはならない）**:
  1. オッズ・EV の入出力は **`data/ev/` にのみ**書く。予想成果物3種・`data/screening/`（選別X軸）には一切流さない。
  2. 確率は**確定済み report.json のエンジン再生（`tools/score_race.py compute_exotics`）のみ**＝オッズを見て確率・印・並びを**変えない**。report.json 未確定のレースでは実行拒否（時系列の壁をコードで強制）。
  3. 出力は「箱候補×点数×的中確率×オッズ×EV」の**表まで**。**金額・Kelly・購入指示は出さない**（買うか・いくら買うかは人間）。
  4. 内在確率は市場較正前＝EV は**参考値**である旨を出力に明記する（較正は results.jsonl の蓄積と `/calibrate-T`・box_sim で改善）。
- 思想: 市場に触れるのは「賭ける瞬間」だけ。予想（X軸）はオッズから独立のまま、その独立性こそが EV の源泉（市場と意見が割れた時だけ妙味が生まれる）。

## I2. 散文%は禁止＋率2カラムの限定例外（論理ファースト）＝**予想レポート（`/analyze-race`）に適用**
- レポート本文の**定性表現・散文に % を書かない**（発揮能力／勝率レンジ／条件付き%／枠別複勝率%… を含め散文%は全廃）。
- 強弱は **印 ◎◯▲△× と行順**で表す。確からしさは**可能性ティア 本線★★★／対抗★★／伏線★**（%でない）。
- **限定例外＝単勝率/複勝率の2カラム**（`rank[]` の `win_prob`/`place_prob`）:
  - **源は `tools/score_race.py` の決定論出力のみ**（手で%を作らない＝恣意性ゼロ）。`rank[]` には **0..1 の数値フィールド**として持ち、web が `×100` して「単勝%」「複勝%」の2カラムを描画する。
  - **文字列フィールドには依然 % を書かない**（散文に「12%」等を埋めない）。率は数値Fのみ・%付与は web 側＝validator は文字列%を引き続き弾く。
  - **並びの主導権は依然 論理（印＋行順＝`rank_order`）**。率は並びに**添える参考列**で、率と並びが食い違ったら**論理が正**（I8）。
  - 較正先（市場）が無い内在%＝**偽の精度**である旨を `header_notes` に明記する。
- 例外2: **選別レイヤー（`/screen-card`）と EVレイヤー（`/ev-board`）は事実ベースの % を出してよい**（オッズ・荒れ条件の基準率＝万馬券率/人気別勝率・EV表の確率カラムなど）。これは予測した確率でなく**実測の市場/統計値**（EV表の確率はエンジン決定論出力）＝偽の精度ではないため。予想本体の散文%禁止とは別物。

## I3. 表中心・最小分量（ユーザーは表しか見ない）
- **作らないもの**: §1 サマリ（「当日の分岐点」1行のみ可）／**mermaid 等の図**／観点別ハイライト。
- 段階フロー（phase_flow）は**1行テキストの因果文**で書く（図にしない）。
- 数字の代わりに **展開列・展開感度・好材料・懸念点** を厚く。各根拠は **[観点タグ A〜K,I] ＋事実＋なぜそう読んだか**。
- 重複を出さない＝読む量を増やさない（結論の正本は §3 着順予想表。他所で繰り返さない）。

## I4. 2成果物の独立
- **成果物1＝展開予想（§2）** と **成果物2＝着順予想（§3）** を**別個に測り・別個に改善**する。
- 展開→着順の**因果は維持**（最有力パターンの phase_flow に各馬を通して並びを論理判断）。
- 各成果物は冒頭に**検証契約**を持ち、レース後に独立採点（展開精度／着順精度の2スコアカード）。

## I5. 展開は複数パターン＋当日可変
- 単一に潰さない。**名前付きパターン**（可能性ティア＋発動トリガー＋pace_level＋脚質別有利不利＋隊列＋phase_flow）で持つ。
- 当日情報（参考R・馬場・乗替・取消）で**ティアを付け替え／1パターンに固定 → 展開感度・並び・展開列を論理で再評価**（素の能力読み＝好材料/懸念点は再調査不要）。

## I6. 確定材料の先取り（§0 後付けにしない）
- 枠順・乗替・回避（取消）が分析時点で確定済みなら、**§2-1/§2-2/§3 本文に最初から織り込む**（脚質への内/外割当て・先行争い再評価まで）。
- §0 当日ボードに残すのは*分析時点で本当に未知のものだけ*（当日馬場・クッション値・含水率・パドック・馬体重・前半参考Rの観察値）。
- 目的: 完成レポートに後付けして**ハーネスを再実行する無駄を避ける**。

## I7. 展開列（箱組みの効率化・成果物2の一部）
- §3 で各馬が**各パターンで圏内になるか・強弱**を一目化（`◎`中心／`○`圏内／`△`条件付き、**圏外は記載なし**。例 `全◎`／`β◎`／`α◎γ◎δ◎ β△`）。
- 源は展開合成の `per_horse_fit`・`leg_advantage`・各パターンの浮上/沈む馬。`predictions.jsonl` の `pattern_fit` にも残す。
- **§3 の直前に転置版「パターン別おすすめ馬」表**（パターン → 中心◎/圏内○/一発△/消し）を置く＝箱組みの逆引き。源は展開列と同一（`per_horse_fit`）で**矛盾させない**（重複でなく転置ビュー＝I3 の例外として許可）。
- 目的は「想定パターン別に無駄なく箱(ボックス)を組む」効率化のみ。**券種・金額・実ベットは人間**（I1 を破らない）。

## I8. エンジン＝率(win_prob/place_prob)の唯一の生成器 ＋ 並びのサニティチェック
- `tools/score_race.py` は **単勝率/複勝率の唯一の源**（I2 の率2カラムの源）＝**毎回回して `rank[].win_prob`/`place_prob`（0..1）に注入する**（旧「任意サニティ」から昇格）。手で%を作らない。
- ただし**着順の並び（`rank_order`・`mark`）は依然 論理ファースト**で決める。率はその並びに**添える参考列**＝率順に並べ替えない。
- **率と論理の並びが食い違ったら 論理側を正**とし、食い違いを点検シグナルとして `header_notes`／`engine_check` に残す（率に合わせて並びを書き換えない）。
- 6ノブ／`PARAMS` の markdown↔code 同値ミラーは従来どおり維持（エンジンが率の源になったので**常時必要**）。

## I9. レポート構造（固定番号）
`§0 当日ボード` → `§1 当日の分岐点(1行)` → `§2 展開予想【成果物1】` → `§3 着順予想表【成果物2】` → `§4 データの確かさ` → `§5 免責`。
- **§2/§3 の番号は他文書（SKILL/各 reference）から参照されるので変えない**。体裁の正本は `skills/analyze-race/references/output-template.md`。

## I11. 散文の内部用語禁止（読者は競馬ファン）
- report.json の**文字列フィールド**（phase_flow の early/mid/late/result、bias、pace_sensitivity、pros/cons の note、engine_check 等）に**エンジン内部の変数名を書かない**。
- 置換表: `A_early`→行き脚 / `A_cruise`→追走力 / `A_finish`→決め手(末脚) / `A_class`→地力 / `phase_flow`(散文中)→段階フロー / `pace_level`(散文中)→ペース水準 / `per_horse_fit`→パターン適合。
- JSON の**キー名**（`"phase_flow":` 等の構造部分）はそのまま。禁止は値の散文のみ。

## I10. 静的データのワンソース化（再調査禁止）
- **不変の事実は正本から一度だけ供給し、web で再調査しない**: コース物理形状（直線・坂・初角・芝スタート）＝`references/course-geometry.md`、種牡馬の産駒傾向＝`references/pedigree-catalog.md`（半静的・年1更新／カタログ外の血だけ web 調査）、出走表スパイン・脚質/テン速・h2h直接対戦・**前走騎手 `recent[].jockey`（K の乗替判定）**＝`tools/fetch_racecard.py` の seed、**斤量・馬格×馬場/芝ダの重量重みづけ＝`tools/weight_adjust.py` の決定論タグ（D/I/G は斤量・馬格を再調査せずこの値を採用。馬場・馬体重は当日に再算定）**、**観点Iの非重量・コース非依存の決定論リスク（高齢/下降基調/大幅昇級/休み明け）＝`tools/risk_flags.py`（I は一次フラグに採用し web は非決定論層＝脚部/気性/故障/中止歴だけ実測。敗因が距離/展開で説明でき地力でないなら割引可）**。
- 供給経路は **spawn 時注入**（SKILL STEP1/3）。subagent は rules を自動ロードしない＝徹底は spawn データと各 agent 定義の記述が担う（誰が何に使うかは各 agent / pace-synthesis 側に書く）。
- 静的データの訂正は**正本ファイルだけ**直す（エコーを作らない）。web で採ってよいのは変動情報（当該開催の馬場・関係者コメント等）のみ。
