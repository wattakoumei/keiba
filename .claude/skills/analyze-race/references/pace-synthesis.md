# 展開合成プロトコル（STEP4a / pace-synthesis）

> `analyze-race` の **STEP4a**。並列ファンアウト（STEP3）が出揃った後に、**1回だけ**走る統合ステップ。
> 展開は「全馬の配置から創発する統合タスク」なので、盲目の並列観点ではなく、**全証拠を突き合わせて**ここで構築する。
> 出力は **複数の名前付き展開パターン**（= analyze-race の成果物1「展開予想」の素）。
> 着順（STEP4b / scoring-model §7-9）はこのパターン群を受け取って算出する。**着順側でパターンを作り直さない。**

## 入力（STEP3 の証拠を突き合わせる）

| 証拠源 | 使う情報 |
|--------|----------|
| 観点 E（展開証拠） | 各馬の脚質（逃/先/差/追）・テン速・想定位置、先行争いの当事者、馬場バイアス、枠の有利不利、コース坂形状 |
| 観点 B（近走） | 近走の実際の脚質・位置取り（Eの脚質読みと突き合わせて確定） |
| 観点 D（適性） | コース/馬場の前残り傾向・内外バイアスの裏付け |
| 観点 K（騎手） | 各騎手の出方（行く/控える）、テン乗りの不確実性、先行意識 |
| 関係者コメント | 「ハナに行く」「今回は控えて」等の戦法宣言（純粋情報・最優先で隊列に反映） |
| 出走表 | 枠順・頭数・距離・馬場状態 |

> **使わない**: オッズ・人気・他人の予想（専門紙の印・予想展開記事）。展開は内在情報のみから組む。

## 矛盾解決ルール

証拠が食い違うときの優先順位：

1. **関係者の戦法宣言 > 推測**。「逃げる」と明言があればその馬を逃げに置く（テン乗りでも宣言を優先）。
2. **近走の実脚質(B) > 一般的な脚質ラベル(E)**。前走で控えた逃げ馬は「控えうる」と扱う。
3. **複数の逃げ/先行馬がいて行く保証がない**＝先行争いが不確実 → これを**別パターンに分岐**する（単一に潰さない）。
4. テン乗り・初距離・初コースの馬の出方は不確実 → その馬が絡むパターンの確率を上げ過ぎない。

## 複数パターンの作り方

**先行争いの分岐点**を軸に 2〜4 個のパターンを立てる。各パターンは「誰がハナを取り、ペースがどうなるか」の
1つの筋書き。確率は「その筋書きが起きる尤度」。

- 典型は **平均/前残り・ハイ/差し台頭・スロー/瞬発** の3系統だが、レースに合わせて命名する（単騎逃げ濃厚なら2個でよい）。
- 各パターンに **発動トリガー**（観測可能な条件。例「⑨が引かずハナ主張」）を必ず付ける → 当日にどのパターンが起きているか判定できる。
- `leg_advantage`（脚質別の有利不利符号 -2..+2）はラップ非依存の主検証軸。レース後に「実際どの脚質が好走したか」と照合する。
- 必要なら馬別に `per_horse_fit`（パターン別の恵まれ度 -2..+2）を付す（枠・位置取りで脚質ラベルから外れる馬用）。
- `Σ prob = 1`。

## 出力スキーマ（PACE_MODEL_SCHEMA）

```json
{
  "patterns": [
    {
      "id": "α",
      "name": "平均ペース・前残り",
      "prob": 0.50,
      "trigger": "⑨が控え単騎④が緩めに逃げる",
      "first_600m": "35.0前後",
      "leg_advantage": {"逃げ": 1, "先行": 1, "差し": -1, "追込": -2},
      "formation_head": [4, 9],
      "formation_last_corner": [4, 2, 9, 5],
      "bias": "前/内有利",
      "per_horse_fit": [{"no": 5, "fit": -1}, {"no": 10, "fit": 1}]
    }
  ],
  "falsification": "⑨がハナ主張すれば β(ハイ)の確率が 0.35→0.6 へ。逆に誰も主張しなければ γ(スロー)上昇。",
  "field_note": "逃げ2・先行3。鍵は⑨の出方。テン乗り⑩は読み難く各パターンで中立寄り。"
}
```

- `formation_head`: 序盤の先頭2〜3頭（馬番）。netkeiba の通過順と照合可能な粒度。
- `formation_last_corner`: 最終コーナーの前方隊列（馬番、前→後）。
- `leg_advantage` と `formation_*` が **ラップが取れなくても着順・通過順から検証できる**指標。

## STEP4b への受け渡し

- scoring-model §7 が `patterns[].leg_advantage` / `per_horse_fit` を `pace_fit` に変換し、各パターンで条件付き着順を出す。
- §8-9 がパターン確率 `prob` で加重して最終着順、各パターンの条件付き着順 `p_i,p` も保持。

## 当日可変（SKILL STEP6 から呼ばれる）

当日情報（前半参考Rで採取したバイアス・パドック・乗り替わり・馬場変化・取消・雨・テン乗り解消など）を受けたら：

**当日バイアスの採取（参考レース選定基準）**: 本命は後半R中心なので、前半の同条件Rを観察して内外・前後のバイアスを採る。
参考Rの一致は **芝/ダート（必須・絶対に混ぜない）＞ 同日・時間帯（必須・直前ほど重い）＞ 回り内外・右左 ＞ 距離帯** の優先順位。
回り・距離は完全一致が理想だが無ければ近いもので代用し**割り引く**（距離違いは決まり手・伸び位置のみ流用）。**1鞍で断定せず複数Rの傾向**で見る（内有利は開催進行で外有利へ反転しうる）。

1. **再調査はしない**（パターン構造は維持）。
2. 当日情報に基づき **パターン確率 `prob` を付け替える**、または「このパターンで確定」と**1つに固定**する。
   - 例: 「⑨が大外枠で楽にハナ→ β(ハイ) を 0.35→0.7」「馬場が急速に乾いて前残り→ α を上げる」。
3. scoring-model §8-9 を**再計算するだけ**で新しい着順を出す（§2-6 の素の力 ability0 はそのまま）。
4. ただし当日情報が**特定馬の状態**（パドック激減・腹目立つ等）なら、その馬の観点 H/G スコアだけ更新して ability0 を部分再計算してよい。

> 当日更新は report.md の §2 に「当日修正」追記、predictions.jsonl には更新後の pace/rank レコードを**追記**（上書きせず履歴を残す。`note` に "当日更新" を付す）。
