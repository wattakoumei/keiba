# review-prediction — 予測精度の検証と重み改善ループ

`/analyze-race` の予測（`data/predictions.jsonl`）と実結果を突合し、
`scoring-model.md` の重み・閾値の改善案を提示する。改善ループの 1 周を担う。
モデル定義は repo ルート `CLAUDE.md`、調整点は
`.claude/skills/analyze-race/references/scoring-model.md`。

## 手順

### STEP 1. 結果の記録

対象レースの確定結果（着順・配当）を web 取得 or ユーザー貼り付けで集め、
`data/results.jsonl` に追記する。1レース1行：

```json
{"race_id":"20260531-tokyo-11","date":"2026-05-31","finish":[{"no":5,"name":"〇〇","pos":1},{"no":8,"name":"△△","pos":2},{"no":3,"name":"□□","pos":3}],"payouts":{"win":{"no":5,"yen":420},"place":{"5":150,"8":260,"3":380},"trio":{"combo":"3-5-8","yen":3120},"trifecta":{"combo":"5-8-3","yen":18400}},"pace_actual":{"first_600m":34.8,"label":"S"}}
```

`pace_actual`（実際の前半ペース・H/M/S）は STEP 1.5 の展開照合に使う。web か映像から取れた範囲で記録。

### STEP 1.5. 展開照合と A/B/C 仕分け（人の目）

`report.md` の §2「展開予想」（脚質分類表・隊列推移・想定ペース・**反証条件**）を、実際のレース
（`pace_actual`・着順・できればレース映像）と照合し、**買い目候補だった馬の外し方を3分類**する。
これは数値集計では出せない因果判定で、改善ループの質を決める。

| 種別 | 判定の目安 | ハーネスへの反映 |
|------|-----------|------------------|
| **A 能力読み違い** | 展開は想定通り。なのに力が足りなかった/上だった | 重み修正の対象（A〜C・状態系） |
| **B 展開読み違い** | 想定ペース/隊列/脚質が外れ、それで着順がブレた | **観点E・適性の修正対象（最重要）** |
| **C 偶然** | 落馬・致命的不利・故障・大出遅れ等、事前に織り込めない事象 | **触らない。母数から除外** |

判定材料は §2 の**反証条件**。「反証条件どおりに崩れた」なら展開想定は妥当＝C寄り、
「反証条件にない崩れ方」なら B。買い目候補の各馬に `A/B/C` タグを付け、results.jsonl に追記：

```json
{"race_id":"20260531-tokyo-11","miss_class":[{"no":5,"class":"C","note":"反証どおりハイ化、想定崩れ＝偶然寄り"},{"no":8,"class":"A","note":"展開向いたが力負け"}]}
```

### STEP 2. 突合・キャリブレーション算出

`predictions.jsonl` の該当レース（複数レース横断も可）と `results.jsonl` を突合し算出：

- **勝率ビン別的中率**: 自分の勝率を 0–10/10–20/… のビンに分け、各ビンの実際の勝率と比較
  （予測 15% のビンが実際 15% 前後なら calibrated）。
- **Brier スコア**: `mean((p_i − 結果_i)^2)`（結果_i は1着なら1, それ以外0）。低いほど良い。
- **回収率 ROI**: 買い目候補（bet_types）の単勝・複勝それぞれで
  `Σ払戻 / Σ賭け金`。三連系を買っていればそれも。
- **妙味の検証**: プラス妙味馬・マイナス妙味馬の実際の成績（市場との優劣が出ているか）。

少数レースでは数値が荒れる点を明記し、サンプル数を併記する。

**C（偶然）の扱い**: STEP 1.5 で `C` と分類した馬は、キャリブレーションと観点バイアス診断の
**母数から除外**する（事前に織り込めない事象なので、重み修正の根拠にすると過学習する）。
ROI は実ベット成績なので C も含めた全体で出すが、「Cを除いた calibration」を併記する。

### STEP 3. 観点バイアス診断

どの観点が系統的に過大/過小評価かを診断する。各観点について、
評価が高かった馬／低かった馬の実際の着順傾向を見て偏りを探す。例：
- 「F 調教を高評価した馬の着順が伸びていない → 調教の重みが過大」
- 「J 市場で過剰人気とした馬が結局好走 → 人気の割引が過剰」
- 「E 展開で恵まれると見た馬が着外続き → 展開読みの精度 or 重み」

観点別に **過大 / 過小 / 妥当** を判定し、根拠（どのレース・どの馬）を添える。

**B（展開読み違い）は観点Eに集約**: STEP 1.5 で `B` と分類した馬は、想定ペース/隊列/脚質の
どれを外したかを集計する。同じ外し方（例: スロー想定→実際ハイが繰り返す）が**複数レースで再発**
したら、観点Eの重み（apt の E 効き幅）や展開予想の手順を見直す根拠にする。単発の B は重みを動かさない。

### STEP 4. 修正案の提示

診断に基づき、`scoring-model.md` の**具体的なパラメータ修正案**を出す：
- base 重み（A/B/C）、apt/cond の効き幅、discount 係数、softmax 温度 T、各 EV/edge 閾値、Kelly 倍率。
- 「T を 14→16 に上げて勝率分布を平坦化（過信を緩和）」のように
  **変更値・理由・期待効果**をセットで。一度に大きく動かさず小刻みに。

### STEP 5. 承認 → 更新（バージョンを上げる）

ユーザー承認を得たら `scoring-model.md` を編集し、
モデルバージョンを上げ（例 v1.0 → v1.1）、「変更履歴」に
バージョン・日付・変更点・理由を追記する。これでループ 1 周完了。

### STEP 6. 回帰確認（任意）

可能なら同じ過去レースを更新後モデルで再分析し、Brier 低下 or ROI 改善の方向かを目視確認。
複数レースが溜まっていれば `predictions.jsonl` / `results.jsonl` から
通算 ROI・通算 Brier をまとめて再集計する。

## 注意

- サンプルが少ないうちは過剰反応しない（1〜2レースで重みを大きく動かさない）。
- 検証は「結果論で印を後付けする」のではなく、calibration と ROI という客観指標で行う。
- 重みは `scoring-model.md` の一箇所のみで管理。複数箇所に散らさない。
