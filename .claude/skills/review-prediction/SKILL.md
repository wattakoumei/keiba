# review-prediction — 予測精度の検証と改善ループ（2スコアカード）

`/analyze-race` の予測（`data/predictions.jsonl`）と実結果を突合し、
`scoring-model.md`・`pace-synthesis.md` の改善案を提示する。改善ループの 1 周を担う。

> **v3.0 の北極星**: 検証対象は **展開予想と着順予想の精度**であって回収率ではない。馬券・市場は持たない。
> **2つの独立スコアカード**で測る：
> - **スコアカード1 展開精度**: 実現パターンの的中・脚質別有利不利・隊列・バイアスの的中（ラップ非依存）。
> - **スコアカード2 着順精度**: 着順相関(Spearman/Kendall)・◎単勝/上位複勝的中率・キャリブレーション・Brier。
>
> 市場（オッズ）は使わないので **モデル vs 市場の比較・α は無い**。
モデル定義は repo ルート `CLAUDE.md`、調整点は `scoring-model.md`（重み）と `pace-synthesis.md`（展開合成手順）。

## 手順

### STEP 1. 結果の記録

対象レースの確定着順を web 取得 or ユーザー貼り付けで集め、`data/results.jsonl` に追記。
**展開の復元に使う通過順・上がりも取れる範囲で記録**する。1レース1行：

```json
{"race_id":"20260531-tokyo-11","date":"2026-05-31","finish":[{"no":5,"name":"〇〇","pos":1,"passing":"3-3","agari":34.5,"style":"差し"},{"no":4,"name":"△△","pos":2,"passing":"1-1","agari":35.2,"style":"逃げ"}],"pace_actual":{"first_600m":34.8,"label":null,"label_reconstructed":"S","reconstructed_from":["勝ち先行②残存","3-4角先頭が確保","上がり最速馬は着外"]}}
```

- `pace_actual.label`（実ラップのH/M/S）が取れなければ `null`。
- **`label_reconstructed`**: ラップが無くても (a)上位馬の脚質（前残りか差し決着か）、(b)3-4角通過順と着順の相関、
  (c)上がり最速馬の着順 から実効ペース層(H/M/S)を**復元**する。`reconstructed_from` に根拠を残す。

### STEP 2. 展開スコアカード（成果物1の採点）— 先に行う

`report.md §2` / `predictions.jsonl` の `record:"pace"` を実結果と照合：

- **実現パターンの特定**: `patterns[].trigger` と実レース（誰がハナを取ったか・隊列）を照合し、どのパターンが起きたかを決める。
- **パターン的中**: 最大確率パターン（argmax prob）が実現パターンと一致したか。＋実現パターンに置けていた確率。
- **脚質別有利不利の的中（主指標・ラップ不要）**: 予測 `leg_advantage` の符号 vs 実際に好走した脚質層（上位3頭の脚質構成）。
- **隊列的中**: `formation_head` vs 実際の3-4角先頭、`formation_last_corner` vs 実通過順。
- **バイアス的中**: 内外/前後予測 vs 実決着傾向。

`results.jsonl` に展開採点を追記（`record:"pace_review"`）：
```json
{"record":"pace_review","race_id":"20260531-tokyo-11","realized_pattern":"α","pattern_hit":true,"leg_advantage_hit":{"逃げ":true,"先行":true,"差し":true},"formation_head_hit":true,"bias_hit":true}
```

### STEP 2.5. A/B/C 自動一次仕分け（展開採点を起点に）

展開スコアカードの結果で、予測した上位馬の外し方を機械的に一次仕分けする（人手は C 認定と境界事例に集中）：

| 種別 | 自動判定 | 反映先（v4.0） |
|------|---------|--------|
| **A 能力読み違い** | **展開的中**（pattern＋脚質有利不利 hit）なのに着順が想定外。かつ**実現パターンの条件付き着順 `conditional[realized]` も外し** | scoring §2 **相別能力マップ**（A_early/cruise/finish/class の配分）・6ノブのうち `w_pos1`(伝達ゲイン)/`T`(較正)・`CLASS_FLOOR` |
| **B 展開読み違い** | 展開を外した（実現パターン≠最大確率 or 脚質有利不利が逆）。または条件付きは合うが**加重で外し**（パターン確率の誤り） | **`pace-synthesis.md`・`pace_level` 写像・観点E手順（最重要）** |
| **C 偶然** | 落馬・致命的不利・故障・大出遅れ等、事前に織り込めない事象（人手認定） | **触らない。母数から除外** |

> **v4.0 の A/B 切り分け**: 実現パターンの**条件付き着順**が当たれば「展開を外しただけ」＝B、条件付きも外せば**相別能力読み**＝A。
> **加古川特別が A の代表例**: 前残りP1を的中させつつ追込◎の条件付き着順も外した＝相別能力の伝達不足。
> 修正先は v3.0 で誤認した「discount 過大」ではなく **A_class 圧縮・`w_pos1` 伝達ゲイン**（v4.0 で是正済み）。
> review は「同型ミスが ≥3レース 再発」して初めて6ノブを1個ずつ小刻みに動かす（単発では動かさない）。

`miss_class` を追記（実データ mizusawa-10 は「展開的中×能力外し」で 5/5=A になる）：
```json
{"race_id":"20260531-tokyo-11","miss_class":[{"no":10,"class":"A","note":"展開(前残り)的中、テン乗り割引の過大で過小評価"}]}
```

### STEP 3. 着順スコアカード（成果物2の採点）

`predictions.jsonl` の `record:"rank"` と実着順を突合（**C は母数から除外**）：

- **着順相関（最重要）**: 予想着順（`predicted_rank` 昇順 or 複勝率降順）と実着順の **Spearman ρ / Kendall τ**。
- **的中率**: ①◎単勝的中率（本命1着）②上位3頭の複勝的中率 ③予想複勝率のビン別的中（キャリブレーション）。
- **Brier**: 1着 `mean((win_prob − 1着_i)^2)`、複勝 `mean((place_prob − 複勝_i)^2)`。
- **★純粋な能力読みの分離計測**: **実現したパターンの条件付き着順**（`conditional[realized_pattern]`）の精度を別途算出。
  これは**ペース誤差を除いた能力読みの精度**。加重着順は当たらなくても条件付きが当たれば「展開を外しただけ」と分かる（=B）。
  逆に条件付きも外していれば能力読み（=A）。**A/B 切り分けの数値的裏付け**になる。

> 少数レースでは数値が荒れる。サンプル数を必ず併記し、1〜2レースで重みを大きく動かさない。

### STEP 4. 観点バイアス診断

各観点について、評価が高/低だった馬の実着順傾向から偏りを探す（C 除外母数で）。例：
- 「F 調教を高評価した馬の着順が伸びていない → 調教の重みが過大」
- 「I で重く割り引いた馬が好走（mizusawa ⑩のテン乗り割引）→ discount が過大」
- 「E 由来の展開で恵まれると見た馬が着外続き → pace_level or 展開合成の精度」
- 「前残り想定で前が残らない／差し想定で差しが届かない → `w_pos1`(伝達ゲイン) or `pace_level` 写像」

観点別に **過大 / 過小 / 妥当** を判定し、根拠（どのレース・どの馬）を添える。
**B が複数レースで再発**（例: スロー想定→実ハイが繰り返す）したら `pace_level` 写像・観点E手順・pace-synthesis.md を見直す根拠に。単発の B は動かさない。

### STEP 5. 修正案の提示（v4.0・6ノブ）

診断に基づき**具体的なパラメータ修正案**を、スコアカード別にフィードバック先を分けて出す：
- **展開スコアカード発（B 系）** → `pace-synthesis.md` の合成・パターン分岐・`pace_level` 写像、`observation-points.md` の観点E手順。
- **着順スコアカード発（A 系）** → `scoring-model.md §2 相別能力マップ`、6ノブ（主に `w_pos1`/`T`、必要なら `CLASS_FLOOR`）。
- 「`w_pos1` 2.6→2.9（前残り想定が複数回で前残らず＝伝達不足）」のように**変更値・理由・期待効果**をセットで、**1ノブずつ小刻みに**。
- **重要**: scoring-model.md の早見表を変えたら `tools/score_race.py` の `PARAMS` を**必ず同値にミラー**する（仕様＝正本、コード＝実行体）。

### STEP 6. 承認 → 更新（バージョンを上げる）

ユーザー承認を得たら該当ファイルを編集し、`scoring-model.md` のモデルバージョンを上げ、
「変更履歴」にバージョン・日付・変更点・理由を追記。pace-synthesis.md を変えた場合もその旨を記す。これでループ 1 周完了。

### STEP 7. 回帰確認（任意）

可能なら同じ過去レースを更新後モデルで再分析し、**展開精度（脚質有利不利の的中）と着順精度（着順相関・Brier）が
別々に**改善方向かを目視確認。複数レースが溜まっていれば通算で2スコアカードを再集計する。

## 注意

- サンプルが少ないうちは過剰反応しない。検証は結果論の印後付けでなく、2スコアカードの客観指標で行う。
- **後方互換**: 旧行（`record` 無し・v1.x/v2.0 の `my_prob`/`edge`/`ev_win`）は「rank 旧形式」とみなし `my_prob→win_prob` のみ読む。
  展開スコアカードは `record:"pace"` を持つ v3.0 以降のレコードでのみ算出（旧レースは pace 母数外）。
  v4.0 行は `phase_abilities` と conditional の `pos/energy/S` を持つ（v3.0 行には無いが、共通キー `win_prob`/`place_prob`/`conditional[].win_prob`/`leg_advantage`/`formation_*` は同じに読める）。
- 重みは `scoring-model.md`（正本）＝`tools/score_race.py` の `PARAMS`（実行体・同値ミラー）、展開合成手順は `pace-synthesis.md` で管理。複数箇所に散らさない。
