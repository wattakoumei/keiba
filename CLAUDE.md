# 競馬予想ハーネス (keiba)

「◯◯レースを分析して」と言うだけで、観点別の並列web調査 → 全証拠の統合を行い、
**展開予想と着順予想を独立した2つの成果物**として出力するための Claude Code ハーネス。
レース結果が出た後に2成果物の精度を別個に検証し、観点・重み・展開合成手順を改善するループを回せる。

> **方針（v3.0〜）**
> - **展開予想と着順予想を独立した2成果物に**（別個に測り、別個に改善）。展開（パターン群）→着順 の因果は維持。
> - **展開は複数パターン＋当日可変**。単一に潰さず名前付きパターン（確率＋発動トリガー＋条件付き着順）で持ち、
>   当日情報でパターン確率を付け替え→着順だけ再計算できる。
> - **市場を完全排除**。オッズ・人気・他人の予想（専門紙の印・予想展開記事）は**証拠にもログにも一切使わない**。
>   予測は馬の内在情報のみで完結する。妙味・EV・Kelly・買い目も無い（**馬券選択は人間が行う**）。

## モデル哲学（展開予想＋着順予想エンジン）

### 1. 能力レイヤー（その馬がどれだけ走れるか）＝ v4.0 相別能力

```
相別能力 = { A_early(序盤・位置取り), A_cruise(中盤・楽な追走), A_finish(終盤・決め手), A_class(相非依存の地力) }
A_class  = 潜在能力 × 適性 × 状態 − 割引 を圧縮した変調乗数（旧 ability0 を支配項から乗数へ降格）
```

- **潜在能力 / 適性 / 状態 / 割引** は従来どおり観点 A〜K から作る（A_class に内包）。
- v4.0 はこれを**相別4値に分解**し、序盤の速さ（A_early）と終盤の決め手（A_finish）を別々に持つ
  ＝逃げ馬と追込馬を単一スカラーで比較しない。観点→相別の対応は `observation-points.md`。
- **偶然**: 点推定を過信しないためのブレ幅（主に展開パターン間のばらつき）。

### 2. 展開レイヤー（ペースが着順をどう作るか）＝ v4.0 相変位再帰モデル

```
展開合成(STEP4a): 全証拠 → 複数の名前付きパターン {id, prob, trigger, pace_level, leg_advantage, 隊列}   # = 成果物1
着順(STEP4b):     相別能力(A_early/A_cruise/A_finish/A_class) を、各パターンの pace_level で
                  3相再帰(序盤→中盤→終盤、状態{pos,energy}を持ち越し) に通して終端スコア S_i,p を出す
                  自分の勝率 p_i = Σ_p (パターン確率_p × softmax(S_·,p)_i)                        # = 成果物2
                  ※計算は決定論ツール tools/score_race.py（手計算しない）
```

- 展開は観点 E 単独でなく、**STEP4a 展開合成**が全証拠（脚質・近走・バイアス・騎手の出方・関係者コメント・枠）を
  突き合わせて作る**複数の確率付きパターン**で表す（pace-synthesis.md）。各パターンは `pace_level`(0..1) を持つ。
- **能力は相別4値に分解**（v4.0）。各パターンの pace_level で3相再帰を回し、位置(pos)と余力(energy)を持ち越して着順を**創発**させる
  → スローは前残り・ハイは差し台頭が**構造から**出る（脚質別有利不利を手で固定しない）。→ Harville で着順分布。
- **各パターンの条件付き着順も保持**。展開の不確実性が着順分布のブレ（偶然レンジ）になる。
- **当日可変**: 当日情報でパターン確率を付け替え／1パターンに固定 → 着順だけ再計算（相別能力は再調査不要）。

> v3.0（単一 ability0 × pace_fit ±15%）は加古川特別で「展開的中・着順外し」の伝達不足を露呈し、v4.0 に置換した（scoring-model.md 変更履歴）。

### 3. 市場は使わない

オッズ・人気・他人の予想は予測にもログにも入れない。モデルは馬の内在情報のみで展開・着順を当てる。
（純度を確保し、予測を群衆＝人気から独立させるため。）

## 使い方

| コマンド | 役割 |
|----------|------|
| `/analyze-race <レース名 or URL or 出走表貼り付け>` | 観点別並列調査→展開合成→ 展開予想・着順予想を出力 |
| `/review-prediction <レース名> <結果>` | 予測 vs 実結果を**展開精度・着順精度の2スコアカード**で検証し改善案を提示 |

データは web 調査が主体。出走表など崩れやすい核データは、取得が不確かなとき
ユーザーに URL 指定または貼り付けでの補強を依頼する（**オッズは取得しない**）。

## 用語

- **観点 (observation point / factor)**: 調査の最小単位。1観点 = 1サブエージェントが担当（既定 10 観点 A〜I, K）。
  カタログは `.claude/skills/analyze-race/references/observation-points.md`。
- **相別能力 (phase abilities)**: 観点を集約した相別の走力推定 A_early/A_cruise/A_finish/A_class（v4.0。旧 ability0 は A_class に内包）。
- **3相再帰 (STEP4b)**: pace_level と相別能力から状態 {pos,energy} を序盤→中盤→終盤と持ち越し着順を創発させる決定論計算（`tools/score_race.py`）。
- **展開合成 (STEP4a)**: 全証拠を突き合わせて複数の展開パターンを作る統合ステップ（pace-synthesis.md）。各パターンは pace_level を持つ。
- **展開パターン**: 名前付きの筋書き（確率＋発動トリガー＋pace_level＋脚質別有利不利＋隊列）。展開予想＝成果物1。
- **着順分布**: パターン加重の 1着率・連対率・複勝率・期待着順＋各パターンの条件付き着順。着順予想＝成果物2。
- **自分の勝率**: パターン加重後の 1着率%。

## データの場所

```
data/
  races/<race-id>/
    出走表.md            # 核データ（レース条件・出走馬。オッズは持たない）
    research-<観点>.md   # 観点別サブエージェントの調査結果
    report.md            # 最終レポート（§2 展開予想 / §3 着順予想の2本柱）
  predictions.jsonl      # 予測ログ（record:"pace" 展開パターン / record:"rank" 着順。市場フィールド無し）
  results.jsonl          # 実結果（着順・通過順・pace_actual+label_reconstructed / pace_review / miss_class）
```

`race-id` は `YYYYMMDD-開催-レース番号`（例: `20260531-tokyo-11`）を推奨。

## 改善ループの回し方

1. `/analyze-race` で予測 → `predictions.jsonl` に pace/rank レコードを追記。
2. レース確定後、`/review-prediction` で `results.jsonl` に結果記録（通過順・復元ペース含む）。
3. **2スコアカード**で検証：①展開精度（実現パターン・脚質別有利不利・隊列の的中）②着順精度（着順相関・的中率・Brier）。
   展開を先に採点して A/B/C を自動一次仕分け（展開的中×着順外し＝A、展開外し＝B、偶然＝C除外）。
4. スコアカード別に修正先を分けて提示：展開発（B）→ pace_fit・観点E・pace-synthesis.md／着順発（A）→ base重み・状態系・discount。
5. ユーザー承認で更新（バージョンを上げる）→ これが 1 周。
