# 競馬予想ハーネス (keiba)

「◯◯レースを分析して」と言うだけで、観点別の並列web調査 → 全証拠の統合を行い、
**展開予想と着順予想を独立した2つの成果物**として出力するための Claude Code ハーネス。
レース結果が出た後に2成果物の精度を別個に検証し、観点・重み・展開合成手順を改善するループを回せる。

> **方針（v3.0〜）**
> - **展開予想と着順予想を独立した2成果物に**（別個に測り、別個に改善）。展開（パターン群）→着順 の因果は維持。
> - **展開は複数パターン＋当日可変**。単一に潰さず名前付きパターン（可能性ティア＋発動トリガー＋段階フロー）で持ち、
>   当日情報でティアを付け替え→展開感度と並びを再評価できる。
> - **市場を完全排除**。オッズ・人気・他人の予想（専門紙の印・予想展開記事）は**証拠にもログにも一切使わない**。
>   予測は馬の内在情報のみで完結する。妙味・EV・Kelly・買い目も無い（**馬券選択は人間が行う**）。
>
> **方針（v5.0・論理ファースト／表中心）**
> - ユーザーは**表しか見ない**。レポートに **% は一切出さない**（発揮能力・1着率・条件付き%等は撤廃）。
> - 着順の並びは**論理**（相変位再帰の因果＝序盤A_early→中盤A_cruise→終盤A_finish）で決め、**印 ◎◯▲△× と行順**で強弱を伝える。
> - 数字の代わりに **展開感度・好材料・懸念点（観点タグ＋なぜそう読んだか）** を厚く書く＝これが判断の根拠。
> - 確定材料（枠順・乗替・回避）は調査段階で揃っていれば本文に**先取り**（§0 後付けにしない＝ハーネス再実行の無駄を避ける）。
> - `tools/score_race.py` は**任意のサニティチェック**（並びの内部整合確認）に降格。% の正本ではない。

- **`.claude/rules/editing-map.md`** — 「何を変えたいか → どのファイルを触るか」のルーティングと、不変則変更時のドリフト防止手順（grep でエコーを追従）。

## モデル哲学（展開予想＋着順予想エンジン）

### 1. 能力レイヤー（その馬がどれだけ走れるか）＝ v4.0 相別能力

```
相別能力 = { A_early(序盤・位置取り), A_cruise(中盤・楽な追走), A_finish(終盤・決め手), A_class(相非依存の地力) }
A_class  = 潜在能力 × 適性 × 状態 − 割引 を圧縮した変調乗数（旧 ability0 を支配項から乗数へ降格）
```

- **潜在能力 / 適性 / 状態 / 割引** は従来どおり観点 A〜K から作る（A_class に内包）。
- v4.0 はこれを**相別4値に分解**し、序盤の速さ（A_early）と終盤の決め手（A_finish）を別々に持つ
  ＝逃げ馬と追込馬を単一スカラーで比較しない。観点→相別の対応は `observation-points.md`。
- **偶然**: 点推定を過信しないためのブレ幅（主に展開パターン間のばらつき）。

### 2. 展開レイヤー（ペースが着順をどう作るか）＝ v4.0 相変位再帰（v5.0 で因果ナラティブとして使用）

```
展開合成(STEP4a): 全証拠 → 複数の名前付きパターン {id, likelihood_tier, trigger, pace_level,
                  leg_advantage, 隊列, phase_flow{序盤→A_early→中盤→A_cruise→終盤→A_finish→結果}}  # = 成果物1
着順(STEP4b):     最有力パターンの phase_flow に各馬を通し、相別能力(A_early/A_cruise/A_finish/A_class)で
                  どの相で浮く/沈むかを論理判断 → 並び（印 ◎◯▲△× と行順）を決める               # = 成果物2
                  ＋各馬を全パターンに通し、各パターンでの圏内/強弱を符号化した「展開列」を付す（想定展開別に箱を効率よく組むための一目表。源は per_horse_fit）
                  ※% は出さない。tools/score_race.py を回した場合は並びの整合チェックにのみ使う
```

- 展開は観点 E 単独でなく、**STEP4a 展開合成**が全証拠（脚質・近走・バイアス・騎手の出方・関係者コメント・枠）を
  突き合わせて作る**複数のパターン**で表す（pace-synthesis.md）。各パターンは `pace_level`(0..1) と **段階フロー `phase_flow`** を持つ。
- **能力は相別4値に分解**（v4.0）。相変位再帰の因果＝「スローは前残り・ハイは差し台頭」を、各パターンの `phase_flow`（序盤の位置取り→中盤の消耗→終盤の決め手）として
  **読める因果ナラティブ**に書き起こす。これが §2 段階フロー（1行テキスト）と §3「展開感度」の素（脚質別有利不利を手で固定しない発想は維持）。
- 報告は確率%でなく**可能性ティア（本線/対抗/伏線）**。着順の並びは論理（印＋行順）で示し % は出さない。
- **当日可変**: 当日情報でティアを付け替え／1パターンに固定 → 展開感度と並びを論理で再評価（相別能力の読み＝好材料/懸念点は再調査不要）。

> v3.0（単一 ability0 × pace_fit ±15%）は加古川特別で「展開的中・着順外し」の伝達不足を露呈し、v4.0 に置換。
> v5.0 は数式を保ちつつ役割を転換：相変位再帰を**並びを論理判断するための因果の語彙**として使い、% 出力を排して表中心レポートにした（scoring-model.md 変更履歴）。

### 3. 市場は使わない

オッズ・人気・他人の予想は予測にもログにも入れない。モデルは馬の内在情報のみで展開・着順を当てる。
（純度を確保し、予測を群衆＝人気から独立させるため。）

## 使い方

| コマンド | 役割 |
|----------|------|
| `/analyze-race <レース名 or URL or 出走表貼り付け>` | 観点別並列調査→展開合成→ 展開予想・着順予想を出力 |
| `/review-prediction <レース名> <結果>` | 予測 vs 実結果を**展開精度・着順精度の2スコアカード**で検証し改善案を提示 |

データは web 調査が主体。出走表など崩れやすい核データは、取得が不確かなとき
ユーザーに URL 指定または貼り付けでの補強を依頼する（**オッズは取得しない**）。

## 用語

- **観点 (observation point / factor)**: 調査の最小単位。1観点 = **1専属 subagent**（`.claude/agents/obs-<id>.md`、`agentType` で並列起動）が担当（既定 11 観点 A〜I, K, L）。
  概念カタログ・相別マッピングは `references/observation-points.md`、全観点共通の規律・スキーマは `references/research-protocol.md`。観点を1つ調整するなら触るのは対応する agent ファイルだけ。
- **相別能力 (phase abilities)**: 観点を集約した相別の走力推定 A_early/A_cruise/A_finish/A_class（v4.0。旧 ability0 は A_class に内包）。v5.0 では並びを論理判断する語彙。
- **段階フロー (phase_flow)**: 各パターンの「序盤→A_early→中盤→A_cruise→終盤→A_finish→結果」の因果文。§2 に**1行テキスト**で記述・§3 展開感度の素（v5.0 で必須。mermaid 図は廃止）。
- **展開感度**: 各馬が「どの展開で浮上/沈むか」を因果で一言（§3 の列）。phase_flow が直接の素。点③⑤の核。
- **好材料 / 懸念点**: §3 の根拠列。各馬2〜4個、**[観点タグ]＋事実＋なぜそう読んだか**で書く（%の代わりの判断根拠）。
- **展開列 (pattern fit)**: §3 着順予想表の列。各馬が各展開パターンで圏内になるかを強弱符号付きで一目化（`◎`中心/`○`圏内/`△`条件付き、圏外は記載なし。例 `全◎`／`β◎`／`α◎γ◎δ◎ β△`）。想定パターンごとに無駄なく箱(ボックス)を組むための早見。源は展開合成の `per_horse_fit`。券種・金額・実ベットは人間判断（買い目・市場は依然として出さない）。
- **可能性ティア**: 展開パターンの確からしさを **本線★★★ / 対抗★★ / 伏線★** で示す（%でなく＝偽の精度を出さない）。
- **3相再帰 (STEP4b の因果骨格)**: pace_level と相別能力から状態 {pos,energy} を序盤→中盤→終盤と持ち越す。v5.0 では並び判断の語彙＝因果。決定論実装 `tools/score_race.py` は**任意のサニティチェック**。
- **展開合成 (STEP4a)**: 全証拠を突き合わせて複数の展開パターンを作る統合ステップ（pace-synthesis.md）。各パターンは pace_level と phase_flow を持つ。
- **展開パターン**: 名前付きの筋書き（可能性ティア＋発動トリガー＋pace_level＋脚質別有利不利＋隊列＋phase_flow）。展開予想＝成果物1。
- **着順予想**: 印 ◎◯▲△× と行順による並び＋各馬の**展開列**（パターン別の圏内/強弱）・展開感度・好材料・懸念点（%は出さない）。成果物2。

## データの場所

```
data/
  races/<race-id>/
    出走表.md            # 核データ（レース条件・出走馬。オッズは持たない）
    research-<観点>.md   # 観点別サブエージェントの調査結果
    report.md            # 最終レポート（§2 展開予想=1行段階フロー / §3 着順予想=展開列・展開感度・好材料・懸念点の表。%・サマリ・mermaid・観点別ハイライトは無し）
  predictions.jsonl      # 予測ログ（record:"pace" 展開パターン+phase_flow / record:"rank" mark・rank_order・pattern_fit・pros/cons。市場フィールド無し）
  results.jsonl          # 実結果（着順・通過順・pace_actual+label_reconstructed / pace_review / miss_class）
```

`race-id` は `YYYYMMDD-開催-RR`（**レース番号は2桁0埋め**。例: `20260531-tokyo-01`）で**統一**。
predictions/results/ディレクトリの突合キーなので揺らさない（`tokyo-1` のような0埋め無しは不可）。

## 改善ループの回し方

1. `/analyze-race` で予測 → `predictions.jsonl` に pace/rank レコードを追記。
2. レース確定後、`/review-prediction` で `results.jsonl` に結果記録（通過順・復元ペース含む）。
3. **2スコアカード**で検証：①展開精度（実現パターン・脚質別有利不利・隊列・**段階フロー**の的中）②着順精度（**並びの順位相関**・◎単勝/上位複勝的中・**好材料/懸念の的中**）。
   展開を先に採点して A/B/C を自動一次仕分け（展開＝phase_flow的中×並び外し＝A、展開外し＝B、偶然＝C除外）。
4. スコアカード別に修正先を分けて提示：展開発（B）→ pace_level・phase_flow・観点E・pace-synthesis.md／着順発（A）→ observation-points.md の相別マッピング・好材料/懸念の読み筋（6ノブはエンジン併用時のみ任意）。
5. ユーザー承認で読み筋を更新 → これが 1 周（エンジンを触った場合のみ scoring-model のバージョンを上げる）。
